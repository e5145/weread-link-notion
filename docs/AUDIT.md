# 旧项目审查与精简说明

旧项目已经证明可以跑通 WeRead 到 Notion 的同步，但长期维护上有几个负担。

## 旧项目中不再迁移的内容

| 内容 | 处理 | 原因 |
|---|---|---|
| Cookie 同步逻辑 | 删除 | 现在已经有 `WEREAD_API_KEY`，不再需要浏览器 Cookie |
| `weread2notionpro` 适配 shim | 删除 | 新版直接实现所需接口，减少猴子补丁和历史包袱 |
| `github-heatmap` 依赖 | 删除 | 新版自己生成 PNG/SVG，避免第三方格式和服务不稳定 |
| `heatmap.malinkang.com` embed | 删除 | 该服务可能解析失败或在 Notion 白屏 |
| 本地 WeRead 探测插件 | 不迁移 | 只用于早期调试，不属于同步器核心功能 |
| `.codex_tmp_*` 缓存 | 不迁移 | 临时解包和测试环境，不应进入正式仓库 |
| `reports/` 探测报告 | 不迁移 | 调试产物，不影响正式功能 |
| 多 workflow 分离 | 合并 | 新版只保留一个 `sync.yml`，顺序更清楚 |

## 新版保留的能力

- 通过 WeRead API Key 获取书架、笔记和阅读统计。
- 自动生成热力图。
- 自动创建 Notion 数据库。
- GitHub Actions 每日自动同步。
- 手动运行和日志排错。

## 新版新增的能力

- 直接生成 `assets/heatmap.png`，Notion 使用原生图片块显示。
- 支持有声书和文章收藏入口的书架总数口径。
- 同步快照数据库记录每次运行结果。
- README 和安装说明按新手流程重写。

## 为什么数据库少了

旧版会创建较多数据库和关系。新版刻意压缩为四个核心数据库：

1. `书库`
2. `笔记`
3. `每日阅读`
4. `同步快照`

这样 Notion 页面加载更快，也更适合移动端。需要复杂分析时，可以在这些数据库上手动加 Notion 视图，而不是让同步器生成过多结构。

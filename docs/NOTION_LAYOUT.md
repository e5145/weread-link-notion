# Notion 页面布局说明

新版布局目标是：

- 首页轻，不堆太多块，打开更流畅。
- 数据库完整，方便筛选、排序、统计和后续自定义视图。
- 自动同步负责基础结构，个性化视图留给你在 Notion 里自由调整。

## 首页会出现什么

第一次同步成功后，`NOTION_PAGE` 指向的页面会追加这些内容：

1. `WeRead Link Notion` 标题。
2. 简短同步说明。
3. 阅读热力图图片。
4. 最近同步预览，包括本次写入数量、最近阅读、最近笔记和最近阅读日。
5. 四个数据库入口。

四个数据库分别是：

```text
书库
笔记
每日阅读
同步快照
```

热力图使用 Notion 原生 image block。图片来自仓库里的：

```text
assets/heatmap.png
```

这样比嵌入第三方热力图网页更稳定，也更适合 Notion 桌面端和移动端。

最近同步预览会在每次运行时自动刷新。它不是完整数据库，只是为了让你打开首页时立刻看到“确实同步进来了”。完整内容仍然保存在下面四个数据库里。

## 书库

保存微信读书书架里的条目，包括：

- 电子书
- 有声书
- 文章收藏入口

主要字段：

| 字段 | 说明 |
|---|---|
| `书名` | Notion 页面标题 |
| `Book ID` | 微信读书唯一 ID |
| `类型` | 电子书 / 有声书 / 文章收藏 |
| `作者` | 作者或演播者 |
| `分类` | 微信读书分类 |
| `状态` | 未读 / 在读 / 读完 |
| `最近阅读` | 最近阅读或收听日期 |
| `私密` | 是否私密 |
| `微信读书链接` | 微信读书网页链接 |
| `划线数` | 这本书的划线数量 |
| `想法数` | 这本书的想法或点评数量 |
| `书签数` | 这本书的书签数量 |

建议手动新增的视图：

- Gallery：按 `状态` 分组，适合浏览书架。
- Table：按 `最近阅读` 倒序，适合查最近读了什么。
- Board：按 `分类` 或 `状态` 分组，适合整理阅读计划。

## 笔记

保存划线和个人想法。

主要字段：

| 字段 | 说明 |
|---|---|
| `内容` | 划线文本或想法正文 |
| `Note ID` | 稳定唯一 ID |
| `Book ID` | 所属书籍 ID |
| `书名` | 所属书籍 |
| `类型` | 划线 / 想法 |
| `章节` | 章节名 |
| `创建时间` | 笔记创建时间 |
| `位置` | 微信读书 range |
| `链接` | 微信读书网页链接 |

建议手动新增的视图：

- Table：按 `创建时间` 倒序。
- Board：按 `书名` 分组。
- Filter：只看 `类型 = 想法`。

## 每日阅读

保存每天的阅读时长。热力图也是基于这个数据生成。

主要字段：

| 字段 | 说明 |
|---|---|
| `日期` | 页面标题 |
| `Date` | Notion 日期字段 |
| `阅读时长(秒)` | 原始秒数 |
| `阅读时长(分钟)` | 分钟数，方便 Notion 图表 |
| `Year` | ISO 年 |
| `Month` | 月份 |
| `Week` | ISO 周 |

建议手动新增的视图：

- Calendar：用 `Date` 字段。
- Chart：X 轴用 `Date`，Y 轴用 `阅读时长(分钟)`。
- Table：按 `Date` 倒序。

## 同步快照

每次同步后写入一条记录，方便排错和观察趋势。

主要字段：

| 字段 | 说明 |
|---|---|
| `同步时间` | 本次同步时间 |
| `Status` | success / failure |
| `Books` | 本次书库条目数 |
| `Notes` | 本次笔记条数 |
| `Read Days` | 有阅读记录的天数 |
| `Read Minutes` | 年内累计阅读分钟数 |
| `Heatmap` | 热力图 URL |
| `Message` | 运行信息 |

建议手动新增的视图：

- Table：按 `同步时间` 倒序。
- Filter：只看 `Status = failure`。

## 为什么不自动创建复杂视图

Notion 官方 API 更适合创建数据库和写入数据，不适合稳定地跨账号创建复杂视图、图表样式和个人偏好的布局。

所以项目采取这个策略：

1. 程序负责稳定创建核心数据库和字段。
2. 你在 Notion 里手动添加自己喜欢的视图。
3. 后续同步只更新数据，不干扰你的自定义视图。

这样 Notion 页面会更流畅，也更不容易因为 API 限制导致布局异常。
